# 新功能描述模板（openspec-viz）

> 加功能前先填这份，**替代"孤立描述一个功能"**。目的：让每个功能锚定「需求交付看板」主线，而不是又长出一个孤岛。先读根目录 [`CLAUDE.md`](../CLAUDE.md)。

## 1. 一句话：这功能让需求交付看板多了什么能力？

> 必须能用"在 ___ 环、给 ___ 角色，看到 / 做到 ___"说清。说不清 = 还没想好它属于主线哪里。

## 2. 两个锚点（硬性，答不上来就别做）

- **旅程环节**：规划 / 设计 / 实施 / 测试 / 上线 —— 哪一环？
- **角色 lens**：PM / QA / 工程 / 跨角色 —— 服务谁？

## 3. 它和中心（requirement）的关系（三选一）

- [ ] 是 requirement 看板上的一个 **lens / facet**（同一份需求数据换看法）
- [ ] 是某 change（需求下单元）里的能力
- [ ] 其它：______ ←（若答案是"新开一个平级页"，**停**——大概率是孤岛，回 `CLAUDE.md` 重想）

## 4. 复用什么（防止重抄、防止漂移）

- **模型**：复用 `shared/types.ts` 的 ______（需新增字段 → 先加这里，不在组件里另造）
- **取数**：`api.ts` 的 ______ / 需新增方法 ______
- **解析**：`parser/` 的 ______（走单点，别新写一套）
- **前端**：`useFetch` / `useLiveEvents` + 复用组件 ______

## 5. 不做什么（划边界）

> 明确这功能"不负责"什么，防止它膨胀成又一个独立产品。

## 6. 验证（业务流程层）

- 从**需求入口**能不能连贯走到它？切到它时是"**换 lens**"还是"跳到另一个独立页"？（后者 = 割裂，回到第 3 步）
- `pnpm typecheck && pnpm test` 绿；`pnpm dev` 手测主流程无断裂。
